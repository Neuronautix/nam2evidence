# NAMO-to-IND Mapper

**A context-of-use-driven evidence packaging tool for NAM-derived nonclinical data.**

NAMO-to-IND Mapper helps drug developers organise, validate, and package evidence generated by New Approach Methodologies (NAMs) into structured dossiers aligned with IND/eCTD submission requirements. The tool does **not** attempt to "convert" NAM data into animal study equivalents — it provides a disciplined workflow for declaring the context of use, documenting validation evidence, building weight-of-evidence claims, and mapping the resulting artefacts to the correct sections of an eCTD nonclinical module.

---

## Key concepts

| Term | Meaning |
|---|---|
| **Context of Use (COU)** | The specific regulatory question and intended use a NAM is deployed to address |
| **NAMO** | New Approach Methodology Ontology — provides controlled vocabulary for model-system classification |
| **Regulatory support level** | Four-tier scale: exploratory → supportive → decision-informing → potentially pivotal |
| **Human review gate** | Every claim starts as `human_review_required`; export is blocked until a qualified reviewer approves all claims |
| **Evidence package** | The complete dossier: COU card + NAMO metadata + validation matrix + WoE claims + eCTD mapping |

## Five evidence workspaces

Every project in NAMO-to-IND Mapper is organised across five interconnected workspaces:

1. **Context of Use (COU) Card** — Declares the NAM type, regulatory question, development stage, intended use, biological domain, population relevance, documented limitations, acceptance criteria, and regulatory confidence level.

2. **NAM Study Metadata** — NAMO-aligned structured record covering: model-system classification (Organoid, OrganOnChip, QSARModel, CellBasedAssay, …), experimental design, assay metadata, data outputs, and full provenance (ELN, LIMS, SOP references, version-controlled analysis scripts).

3. **Validation Evidence Matrix** — Eight-domain framework assessing model fitness-for-purpose: analytical validity, technical reproducibility, biological relevance, reference compound performance, exposure relevance, data integrity, limitation analysis, and regulatory alignment. Each criterion carries a status (met / partial / not met / N/A).

4. **Weight-of-Evidence Claim Graph** — Hierarchical tree of structured claims linking evidence to regulatory conclusions. Claims are typed (mechanistic / empirical / comparative / predictive) and carry one of the four regulatory support levels. A human-review gate must be cleared before export is enabled.

5. **eCTD Module 4 Mapper** — Maps evidence documents and claim summaries to the appropriate sections of an IND/eCTD nonclinical module (e.g. 4.2.3.7.3 for other in vitro studies, 2.6.6 for toxicology written summary) with placement justification.

## Getting started

### Prerequisites

- Docker Desktop ≥ 24 (with Compose v2)
- Node.js 20+ (for local frontend development without Docker)

### Run with Docker Compose

```bash
git clone https://github.com/Neuronautix/NAMO-to-IND-Mapper.git
cd NAMO-to-IND-Mapper

# Copy and edit the API environment file
cp api/.env.example api/.env

# (Optional) Copy the frontend env example if you want to override the API URL
cp frontend/.env.example frontend/.env.local

# Start all services
docker compose up --build

# Apply database schema (required on first run)
docker compose exec -T api php bin/console doctrine:migrations:migrate --no-interaction
```

The frontend runs in API mode by default via `NEXT_PUBLIC_DATA_MODE=api`.

### Environment variables

| File | Purpose |
|---|---|
| `api/.env.example` | Template for Symfony env vars (`APP_ENV`, `APP_SECRET`, `DATABASE_URL`, `CORS_ALLOW_ORIGIN`). Copy to `api/.env`. |
| `frontend/.env.example` | Template for Next.js env vars (`NEXT_PUBLIC_API_URL`). Copy to `frontend/.env.local`. |

`docker-compose.yml` reads `APP_SECRET` from the host environment when set, falling back to a development default. To override it for production:

```bash
APP_SECRET="$(openssl rand -hex 32)" docker compose up -d
```

For hot-reload frontend development inside Docker, copy `docker-compose.override.yml.example` to `docker-compose.override.yml` — Compose will merge it automatically.

| Service | URL |
|---|---|
| Frontend (Next.js) | http://localhost:3000 |
| Backend API | http://localhost:8080/api |
| API documentation | http://localhost:8080/api/docs |

### First-use check

After first startup and migration:

- Open the frontend: http://localhost:3000
- Verify backend health by opening: http://localhost:8080/api/v1/projects
- Expected response in a fresh environment: `[]`

### Run the frontend locally (standalone demo)

The frontend ships with complete demo data (liver organoid hepatotoxicity use case) and works without a running backend.

```bash
cd frontend
npm install
npm run dev
# open http://localhost:3000
```

To force demo mode locally, set `NEXT_PUBLIC_DATA_MODE=demo`.

To run against the backend API, set:

```bash
NEXT_PUBLIC_DATA_MODE=api
NEXT_PUBLIC_API_URL=http://localhost:8080
```

### Run the backend locally

```bash
cd api
composer install
cp .env.example .env   # edit DATABASE_URL
php bin/console doctrine:migrations:migrate
symfony server:start
```

## How to use the app

### Typical user flow

1. Create a project on the dashboard.
2. Complete the Context of Use workspace.
3. Enter NAM Study metadata.
4. Fill the Validation Evidence Matrix.
5. Build and review Claim Graph claims.
6. Add eCTD mapping entries.
7. Go to Export Center and download artifacts.

### Export behavior

- Export generation is backend-driven for all formats.
- A server-side review gate blocks export while any claim is `human_review_required`.
- Blocked exports return an error with pending claim IDs.
- Successful exports persist an export package snapshot for audit/history.

## Functionality and behavior summary

### What the app does today

- Project creation and listing.
- Project-scoped workspace retrieval from backend APIs.
- COU update and evidence status updates through API endpoints.
- Claim status updates through API endpoints.
- Project-scoped export package generation.
- Backend-generated downloads from Export Center in JSON, CSV, Markdown, and TXT formats.

### API/data mode behavior

- `NEXT_PUBLIC_DATA_MODE=api`: frontend uses backend endpoints (recommended for normal use).
- `NEXT_PUBLIC_DATA_MODE=demo`: frontend uses bundled demo data without backend.

## Regulatory support levels

| Level | Colour | Meaning |
|---|---|---|
| **Exploratory** | 🟡 Yellow | Hypothesis-generating; not sufficient for regulatory decisions alone |
| **Supportive** | 🔵 Blue | Adds weight to a body of evidence; suitable for IND narrative context |
| **Decision-informing** | 🟢 Green | Sufficient to inform specific regulatory decisions in defined COU |
| **Potentially pivotal** | 🟣 Purple | Could replace a traditional study in a specific COU with regulator agreement |
